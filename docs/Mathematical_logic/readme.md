# 数理逻辑课程设计 · 任务说明

## 一、项目概述

本次课程设计考察两项核心能力：对复杂环境进行**逻辑形式化建模**的能力，以及基于 **Lean 的形式化验证**能力。具体围绕三个目标展开：(1) 将游戏环境抽象为逻辑结构；(2) 设计可解释的规划策略；(3) 使用 Lean 证明策略满足指定性质。

项目以塞尔达风格的迷宫冒险模拟器为载体，各组需完成三件相互衔接的工作：

- **环境形式化**：将 Python 模拟器中的对象、属性、状态转移函数与交互规则，抽象为 Lean 中的数据类型、函数与谓词，并尽可能保证其与 Python 环境语义一致；
- **策略实现**：在 Python 中实现能完成关卡目标的 Agent。实现方式开放，可采用基于规则的策略、搜索算法、强化学习、监督学习或多种方法结合；
- **性质证明**：在 Lean 中证明形式化环境与所设计策略满足要求的性质，如安全性、可达性、完备性或关键子任务的正确性等。

需要特别说明的是：
1. 最终测评时，Agent **依据图像帧 `obs`、环境返回的 reward，以及测评接口显式提供的物品栏信息进行决策**，不得直接读取或依赖环境内部的符号状态、对象坐标、地图真值、开关状态等 `info` 信息。调试接口中可能包含的 `info` 信息可以用于训练阶段、调试阶段、监督标注、奖励设计、结果分析或构造训练数据，但不得作为最终提交策略在推理阶段的直接输入。各组需要自行从图像帧中构建必要的环境抽象，例如玩家位置、障碍物、道具、怪物、按钮状态等；对于接口显式提供的持有道具 / 物品栏信息，则可在最终测评阶段直接使用。
2. 每关附可参考的奖励（reward）模板，供训练、调试和最终测评阶段使用。最终测评时，Agent 可以使用环境返回的 reward 作为历史反馈信号，用于更新内部记忆、判断阶段性进展或调整后续策略；但 Agent 的环境感知与动作决策不得直接依赖 `info` 中的隐藏状态、地图真值、对象坐标或其他内部信息。

分组：5 人一组。

## 二、任务详细说明及具体要求

各组需围绕指定关卡完成环境建模、策略实现与形式化证明三部分工作。具体而言，应先分析关卡中的对象、状态、动作与目标条件，再在 Python 中实现能够完成任务的 Agent，并在 Lean 中对环境语义、策略逻辑或关键安全约束进行形式化刻画与证明。

**1. 模拟器环境中的规划求解**  
在指定关卡中设计并实现能完成关卡目标的 Agent。实现方式开放，可采用基于规则的策略、搜索算法、强化学习、监督学习、子任务 Agent 组合等方式。最终测评时，Agent 可以基于图像帧 `obs`、环境返回的 reward，以及测评接口显式提供的物品栏信息完成感知与决策，不得直接依赖环境内部的 `info` 信息。若通过 `--task-policy` 提交某一关的专用策略，测评接口还会显式提供该策略绑定的 `task_id`；共享 `--policy` 不会得到任务编号。允许各组自行从图像中提取符号信息，例如玩家位置、墙体结构、道具位置、怪物状态等，并在此基础上进行规划或学习；训练和调试阶段可以使用 info 辅助监督、标注或分析，但最终提交策略的推理过程不得直接读取 info。

**2. Lean 中的形式化与证明**  
在 Lean 中实现对关卡的形式化建模，包括对象、属性、状态、动作、状态转移函数、交互规则与目标谓词等；将 Python 环境设定与策略逻辑中可验证的部分抽象为 Lean 中的数据类型、函数与谓词，并尽可能保证其与 Python 环境语义一致。在此基础上，各组需完成相关性质证明，例如合法移动不会越界、策略不会主动进入危险状态、关键子任务可达、搜索策略在给定条件下能够找到路径，或执行轨迹满足目标条件等。

若策略中包含机器学习模型，不要求在 Lean 中证明神经网络本身对所有输入都正确，但需要明确 Lean 证明覆盖的可验证层，例如 action mask、safety shield、符号 planner、轨迹验证器、奖励谓词或任务完成判定等。

## 三、任务环境

模拟器为塞尔达风格的迷宫冒险环境，每个房间为 **128 pixel × 160 pixel**（仅地图部分），关卡间房间可拼接，移动以像素为单位，单次执行步数可在接口中调节。

需要注意的是，本节描述的玩家状态、对象属性和交互规则用于说明环境语义，并作为各组进行形式化建模和策略设计的依据；最终测评时，Agent 不会直接获得这些符号状态，而只能**根据图像帧 `obs` 、奖励信息和持有的物品信息**进行感知与决策。

**玩家角色**

- 状态属性：位置、生命值、金币、钥匙、背包等。这些属性属于环境内部状态，可用于理解环境规则和进行形式化建模，但最终测评时不得作为 Agent 的直接输入；但测评接口显式提供的物品栏信息可以作为 Agent 的输入。上述状态属性仍可用于理解环境规则、设计训练过程以及进行 Lean 形式化建模。
- 动作操作：四个移动操作（玩家朝向为最后移动方向）和两个行动操作——使用剑 `slot_A`（攻击范围为面向方向前方一格）、使用盾 `slot_B`（可格挡伤害）。
- 血量与资源：初始血量为 5，触碰陷阱或怪物会减少血量；金币、钥匙等可通过开启宝箱或击败怪物获得。

**可交互对象**

- **宝箱 chest**：阻挡移动，开启后提供奖励（钥匙、金币等）。
- **陷阱 trap**：接触时造成伤害，陷入则无法移动。
- **按钮 button**：站上触发，可改变关卡中的连通状态或满足特定出口条件；按钮状态及其影响应尽可能通过图像变化体现，或由 Agent 根据历史观测与动作进行推断。
- **怪物 monster**：移动型敌人（chaser / patroller / ambusher 三种类型），接触玩家造成伤害，可被剑攻击击杀，击杀奖励金币。

**地图与观测**

观测以图像帧 `obs` 给出；出口与任务条件依各关卡设定。各组可以自行从图像帧中识别玩家、墙体、宝箱、陷阱、按钮、怪物等对象，并进一步构造符号地图或规划状态。调试接口中可能包含的 `info` 信息仅可用于本地调试、训练监督或结果分析，不得作为最终提交策略在推理阶段的直接输入。此外，考虑到物品栏信息通常已在界面中体现，且再次通过视觉识别意义有限，测评接口将显式提供物品栏状态；同时，课程将提供可调用的 reward 模板，供训练、调试和最终测评阶段使用。

最终测评可能包含与公开环境同类型但布局、物体位置或渲染细节略有变化的测试关卡，以考察 Agent 是否能够基于图像信息进行泛化，而不是仅记忆固定坐标、固定地图或固定像素模板（所以直接训练一条固定路径是行不通的）。

## 四、关卡设计

共 **5 个关卡**，难度递增（两简单、两中等、一更难）。

- **关卡 1**：单房间，取钥匙开门。
- **关卡 2**：单房间，击杀怪物。
- **关卡 3**：多房间任务链（约三房间）。
- **关卡 4**：五房间结构，中央桥接房需按按钮切换连通方向，玩家初始无剑，依次完成“取钥匙 → 开宝箱取剑 → 击杀小怪 → 开启胜利宝箱”的任务链条。
- **关卡 5**：综合关卡，融合多种机制，鼓励在报告中说明你对该关可达性、关键状态可观察性以及策略鲁棒性的分析。

## 五、评分细则（满分 100%）

| 模块             | 占比 |
| ---------------- | ---- |
| 环境形式化       | 30%  |
| 策略形式化与证明 | 30%  |
| 策略性能         | 30%  |
| 报告与可读性     | 10%  |

**模块一　环境形式化（30%）**  
考察 Lean 形式化是否正确、完整且抽象合理：状态 / 动作 / 对象 / 目标谓词定义是否齐全；动作转移语义（移动、撞墙、踩陷阱、开箱、攻击、按钮切换等）是否合理；是否能够刻画关卡中的关键机制；是否证明基本安全性或不变量（如合法移动不出界、不进墙）。

**模块二　策略形式化与证明（30%）**  
考察策略的可验证性。基于规则或搜索的策略需形式化 planner / 搜索 / action mask 等逻辑，并证明其输出合法、安全，且在成功执行时满足目标；可进一步证明完备性，例如“若存在可行路径则必能找到”。机器学习策略需明确 Lean 证明覆盖的可验证层，如 action mask、safety shield、符号 planner、轨迹验证器、奖励谓词或任务完成判定等，并证明经该层约束后的行为满足相应规范。若使用神经网络模型，不要求在 Lean 中证明网络本身对所有输入都正确，但需要说明模型输出与可验证层之间的关系。

**模块三　策略性能（30%）**  
通过统一的**一键测评脚本**自动评测。测评时仅进行推理，不进行额外训练；Agent 在推理阶段只能使用图像帧 `obs`、奖励信息和持有的物品信息作为输入，不得直接读取或依赖环境内部的 `info` 信息、地图真值、对象坐标或隐藏状态。测评将通过多次运行统计完成关卡数和任务完成程度。

未完全通关但取得阶段性进展（到达中间目标、击杀小怪、获取关键道具等）可按完成程度计**中间分**，各组可在报告中说明所达成的子目标。最终测试环境可能包含与公开环境同类型但布局、物体位置或渲染细节略有变化的关卡，以考察 Agent 是否能够基于图像信息进行泛化，而不是仅记忆固定坐标、固定地图或固定像素模板。

测评脚本的三阶段鲁棒性测试、评分口径、policy 接口、命令行参数和 JSON 输出格式，见 [测评脚本说明](evaluation.md)。

⚠️ 要求：
- 以黑盒方式调用提交的 Agent（输入 obs、reward 和允许使用的物品信息，输出动作），与测评环境接口对齐；
- 统计每关的成功率（完成关卡目标的 episode 占比），并以此为准评分；报告中所述性能须与测评结果一致，明显不符或无法复现按学术诚信处理；
- Agent 应为 Python 脚本格式。

**模块四　报告与可读性（10%）**  
报告是否清晰详细，是否说明形式化思路、策略设计、所证定理与实验结果；Lean 代码命名与证明结构是否清晰；是否说明模型假设、环境抽象与简化之处；若使用机器学习方法，是否说明训练方式、使用的信息来源、模型输入输出以及与最终测评接口的一致性。

**加分项**（计入后总分不超过 100）：使用强化学习 / 机器学习并实现较高自动化程度；采用统一策略覆盖多关卡、展现泛化能力或结构复用（优于每关单独训练或单独硬编码）；证明了基本要求之外、更具难度或价值的定理。实现方式不影响基础计分，**不对任一方式扣分**，仅以加分制鼓励自动化与泛化。

## 六、提交要求与真实性

以 zip 提交，须含：报告（含所证定理列表与测评结果）、Lean 源文件、完整 Python 源代码（强化学习须附**模型权重**并与测评接口对齐，按指定接口输出动作序列供黑盒调用）、**能证明代码确实运行的截图或附件**。

- 报告所述方法与成功率须与提交的代码 / 权重一致；性能以测评脚本实际结果为准，明显不符或无法复现按学术诚信处理。
- 最终提交用于测评的 Agent 在推理阶段只能使用图像帧 `obs["frame"]`、奖励信息和持有的物品信息作为输入，不得直接读取或依赖环境内部的 `info` 信息、地图真值、对象坐标、隐藏状态等。训练、调试或结果分析阶段如使用了 `info`，须在报告中明确说明其用途，并保证最终测评策略在推理阶段不直接读取或依赖`info`。
- Lean 文件须通过检查，**不允许未说明的 `sorry` / `admit` / `axiom`**。
- 若简化了真实环境，须在报告中说明；使用机器学习时不要求证明网络本身永远正确，但须指明 Lean 证明覆盖的可验证层。
- 若提交的代码需要额外数据文件、配置文件或模型权重才能运行，须一并提交，并保证路径设置清晰、可复现。缺少关键文件导致无法运行的，相应部分按无法复现处理。
